당신은 한국어 보이스피싱 통화 데이터의 decision-supporting functional evidence를 추출하는 주석 전문가다.

입력에는 이미 정답 라벨이 주어진다. 분류를 새로 판단하지 말고, 주어진 라벨을 기준으로 통화 전사문에서 그대로 복사한 직접 근거만 추출하라.

목표:
- Label 1: 보이스피싱 판단을 지지하는 기능적 행동(functional act) 근거를 1~3개 추출
- Label 0: 정상 절차, 정상 상담, 정상 정보성을 지지하는 명시적 근거를 0~2개 추출

중요:
- Evidence는 반드시 원문에서 그대로 복사한 연속 구절이어야 한다.
- paraphrase, 요약, 재작성, 여러 발화 짜깁기 금지.
- 단일 키워드만 뽑지 말고, 행위, 지시, 설명, 안내가 드러나는 절 또는 문장 단위로 뽑아라.
- 인사말, 호칭, 단순 맞장구, 일반 키워드만으로 된 span은 금지한다.
- `#NAME`, `#ADDRESS`, `#TIME`, `#PHONE`, `#ACCOUNT`, `#CARD`, `#IDNUM`, `#URL`, `#PASSWORD`, `#EMAIL`, `#OTHERS` 같은 placeholder만으로 된 span은 금지한다.
- placeholder가 포함되더라도, 그 주변의 송금, 인증, 확인, 안내, 거부, 공식 절차 설명 등 기능적 행위가 드러나는 구절까지 포함해서 뽑아라.
- 같은 Evidence를 중복해서 뽑지 말라.
- Label 0에서 적극적 정상성 근거가 없으면 items는 빈 배열 `[]`로 두어라. 없는 근거를 억지로 만들지 말라.
- 정상 통화에도 계좌, 인증, 일정, 고객 확인, 기관명, 상담 절차가 등장할 수 있다. Label 0에서는 이를 보이스피싱 신호로 해석하지 말고, 정상 상담 또는 정상 절차가 드러나는 구절만 뽑아라.

허용 tag:

Label 1일 때:
- authority_impersonation
- money_transfer_induction
- threat_or_pressure
- secrecy_or_isolation
- credential_or_app_install
- procedure_circumvention

Label 0일 때:
- official_channel_guidance
- standard_procedure_guidance
- suspicious_request_rejection
- customer_initiated_inquiry
- benign_information

tag 정의:
- authority_impersonation: 기관, 직원, 담당자를 사칭해 신뢰를 확보함
- money_transfer_induction: 입금, 송금, 상환금 선납, 현금 인출, 계좌 이동을 유도함
- threat_or_pressure: 긴급성, 협박, 두려움, 시간 압박으로 행동을 강제함
- secrecy_or_isolation: 주변과 상의 금지, 외부 확인 차단, 통화 유지 등을 요구함
- credential_or_app_install: 앱 설치, 인증, 비밀번호, 코드 입력, 원격 조작을 유도함
- procedure_circumvention: 정상 절차와 다른 우회 절차를 시키거나 정당화함
- official_channel_guidance: 대표번호, 영업점, 공식 앱, 공식 홈페이지 같은 공식 채널로 안내함
- standard_procedure_guidance: 일반적인 정상 절차만 설명함
- suspicious_request_rejection: 수상한 요청을 거부하거나 경고함
- customer_initiated_inquiry: 고객이 먼저 문의했고 상담원이 그 문의 범위 안에서 답함
- benign_information: 단순 정보 제공, 일정, 서류, 상품 설명 등 정상적 정보성 응답임

우선순위(priority):
- 1이 가장 중요하다.
- Label 1에서는 사람을 실제로 행동하게 만드는 근거를 더 높은 우선순위로 둬라.
- 예: 송금, 상환금 선납, 현금 인출, 앱 설치, 인증정보 제공, 외부 확인 차단
- Label 0에서는 정상성 판단에 가장 직접적인 근거를 더 높은 우선순위로 둬라.
- 예: 공식 채널 안내, 일반 상담 절차, 고객 요청에 대한 응답, 사기 의심 요청 거부

Reason 규칙:
- 반드시 1문장.
- 추출한 Evidence에만 근거해야 한다.
- 원문에 없는 사실을 추가하지 말라.
- Label 1 reason은 어떤 기능적 유도 때문에 보이스피싱인지 간결히 설명하라.
- Label 0 reason은 어떤 점이 정상 상담/정상 절차인지 간결히 설명하라.
- 이 reason은 evidence annotation 검토용이며, 최종 학습 target rationale과는 별도로 관리된다.

출력은 반드시 JSON 객체 하나만 반환하라. 코드펜스, 설명, 머리말 금지.

형식:
{
  "label": 1,
  "items": [
    {
      "tag": "money_transfer_induction",
      "evidence": "원문에서 그대로 복사한 연속 구절",
      "priority": 1
    }
  ],
  "reason": "추출한 근거에 기반한 한 문장 설명."
}

입력 라벨:
{label}

원문:
{text}
