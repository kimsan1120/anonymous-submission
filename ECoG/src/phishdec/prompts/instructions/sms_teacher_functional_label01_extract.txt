당신은 한국어 스미싱 문자 데이터의 functional evidence를 추출하는 주석 전문가다.

입력에는 이미 정답 라벨이 주어진다. 분류를 새로 하지 말고, 주어진 라벨을 기준으로 문자 원문에서 그대로 복사한 직접 근거만 뽑아라.

목표:
- Label 1: 스미싱을 성립시키는 기능적 행동(functional act) 근거를 1~3개 추출
- Label 0: 정상 안내/정상 절차/정상 정보성의 적극적 근거를 0~2개 추출

중요:
- Evidence는 반드시 원문에서 그대로 복사한 연속 구절이어야 한다.
- paraphrase, 요약, 재작성, 여러 구절 짜깁기 금지.
- 단일 키워드만 뽑지 말고, 유도/요청/안내/경고/확인 행위가 드러나는 구절 단위로 뽑아라.
- URL, 전화번호, 인증번호, 계좌번호, 운송장번호, 날짜/시간 placeholder만 단독으로 뽑지 말라.
- `#URL`, `#PHONE`, `#PASSWORD`, `#ACCOUNT`, `#TRACKING`, `#IDNUM`, `#EMAIL`, `#NAME`, `#ADDRESS`, `#TIME`, `#OTHERS` 같은 placeholder만으로 된 span은 금지한다.
- 링크나 전화번호가 있더라도, 그 주변의 클릭/문의/신청/수정/신고/확인 유도 문구까지 포함해서 뽑아라.
- 같은 Evidence를 중복해서 뽑지 말라.
- Label 0에서는 정상성의 적극 근거가 없으면 items는 빈 배열 `[]`로 두어라. 없는 근거를 억지로 만들지 말라.
- 정상 문자에도 URL, 대표번호, 배송조회 링크, 이벤트 링크, 결제/가입 안내가 들어갈 수 있다. Label 0에서는 이를 스미싱 신호로 해석하지 말고, 정상 안내 기능이 드러나는 구절만 뽑아라.
- 광고 표기, 브랜드명, 택배사명, 은행명 자체만으로 근거를 만들지 말라. 실제 기능이 드러난 구절을 뽑아라.

허용 tag:

Label 1일 때:
- authority_or_brand_impersonation
- benefit_or_loan_lure
- payment_or_transaction_claim
- delivery_issue_claim
- credential_or_document_request
- link_or_contact_induction
- urgency_or_deadline_pressure

Label 0일 때:
- routine_delivery_notice
- official_service_notice
- marketing_or_event_notice
- customer_requested_transaction_notice
- security_or_fraud_warning
- benign_information

tag 정의:
- authority_or_brand_impersonation: 공공기관, 금융기관, 택배사, 통신사, 서비스센터, 브랜드 담당자 등을 내세워 신뢰를 확보하거나 그 명의로 행동을 유도함
- benefit_or_loan_lure: 정책자금, 생활안정자금, 저금리 대출, 채무통합, 지원대상 선정 등 혜택/대출 가능성을 미끼로 반응을 유도함
- payment_or_transaction_claim: 해외결제, 주문완료, 미납, 장기체납, 세금, 승인, 결제오류 등 금전/거래 상황을 제시하며 대응을 유도함
- delivery_issue_claim: 주소 오류, 배송 보류, 반송, 미수령, 재배송 요청 등 배송 문제를 제시하며 조치를 유도함
- credential_or_document_request: 인증번호, 신분증, 개인정보, 계좌, 서류, 본인확인정보 제출이나 회신을 요구함
- link_or_contact_induction: 링크 클릭, 전화 문의, 1:1 상담, 답장, 카카오톡/문자 회신 등 추가 접촉 행동을 유도함
- urgency_or_deadline_pressure: 당일 마감, 즉시 문의, 기간 내 신청, 조기 소진, 미조치 시 불이익 등 시간 압박이나 긴급성을 부여함
- routine_delivery_notice: 일반적인 배송출발, 배송완료, 배송예정, 기사안내, 배송지연 안내 등 정상 배송 상태를 전달함
- official_service_notice: 은행, 통신사, 공공기관, 서비스 제공자의 일반 공지, 서비스 안내, 정책 안내, 장애/점검/발급 안내 등 정상 운영 정보를 전달함
- marketing_or_event_notice: 광고, 프로모션, 쿠폰, 이벤트 참여, 혜택 안내 등 정상 마케팅/홍보성 정보를 전달함
- customer_requested_transaction_notice: 사용자가 실제 신청/가입/주문/문의한 거래 또는 그 후속 처리 결과를 안내함
- security_or_fraud_warning: 메신저피싱, 사칭, 금전요구 등 사기 예방 또는 보안 주의사항을 경고함
- benign_information: 단순 정보 제공, 일정 안내, 일반 고지, 고객 응대 등 정상적 정보성 내용임

우선순위(priority):
- 1이 가장 중요하다.
- Label 1에서는 실제 행동을 유도하는 근거를 더 높은 우선순위로 둬라.
- 예: 링크 클릭, 전화 문의, 개인정보 제출, 주소 수정, 신청 마감, 결제 대응 유도
- Label 0에서는 정상 거래/정상 절차를 가장 직접적으로 보여주는 근거를 더 높은 우선순위로 둬라.
- 예: 배송 완료/예정 안내, 고객 신청 결과 안내, 공식 공지, 예방 안내

Reason 규칙:
- 반드시 1문장.
- 추출한 Evidence에만 근거해야 한다.
- 원문에 없는 사실을 추가하지 말라.
- Label 1 reason은 어떤 기능적 유도 때문에 스미싱인지 간결히 설명하라.
- Label 0 reason은 어떤 점이 정상 안내/정상 절차인지 간결히 설명하라.

출력은 반드시 JSON 객체 하나만 반환하라. 코드펜스, 설명, 머리말 금지.

형식:
{
  "label": 1,
  "items": [
    {
      "tag": "delivery_issue_claim",
      "evidence": "배송주소 오류로물품이 반송 되었습니다문의를 통해 주소를 수정해주세요#URL",
      "priority": 1
    },
    {
      "tag": "link_or_contact_induction",
      "evidence": "문의를 통해 주소를 수정해주세요#URL",
      "priority": 2
    }
  ],
  "reason": "배송 문제를 제시한 뒤 링크를 통해 추가 조치를 유도한다."
}

입력 라벨:
{label}

원문:
{text}
